## ğŸ¯ The Paradigm Shift

### Traditional Model

Traditional product teams are built around **specialized roles**:

| Role | Focus |
|------|-------|
| Software Engineer | Writes code |
| Designer | Creates UI/UX |
| Product Manager | Defines what to build |
| Engineering Manager | Develops engineers |
| Delivery Lead | Ensures teams ship |

This model scales by adding specialists. A team of 8-10 people might have 5 engineers, 1 designer, 1 PM, 1 EM, and a shared delivery function. Coordination overhead grows with headcount.

### AI-Native Model

AI tooling changes the economics. A single person with strong judgment across multiple domains can now leverage AI to do work that previously required specialists. The constraint shifts from **hands to do the work** to **judgment to direct the AI**.

The Builder model replaces specialization with **capable generalists**:

| Traditional Roles | Replaced By |
|-------------------|-------------|
| Software Engineer | Builder (Technical Judgment + AI Orchestration) |
| Designer | Builder (Design Judgment) |
| Product Manager | Builder (Product Judgment + Commercial Awareness) |
| Engineering Manager | Master Builder (develops Builders) |
| Delivery Lead | Embedded in Builder accountability |

### Implications

- **Smaller teams, higher output:** 2-3 Builders can deliver what previously required 8-10 specialists
- **Generalists over specialists:** Breadth of judgment matters more than depth in one domain
- **Development model changes:** Apprentices learn all six capabilities, not a single discipline
- **The constraint is judgment:** You can only scale as fast as you can develop people with cross-domain judgment

## ğŸ¯ Core Concepts

### Three Tiers

| Role | Description | Development Responsibility |
|------|-------------|---------------------------|
| **Master Builder** | Exceptional across all capabilities. Owns architectural coherence. The constraint on scale. | Develops Builders |
| **Builder** | Strong in most capabilities. Executes autonomously with AI tools. Knows when to escalate. | Develops Apprentices |
| **Apprentice** | Developing judgment under direct guidance. Learning to evaluate AI output critically. | Learning |

### Six Capabilities

| Capability | What "Strong" Looks Like |
|------------|-------------------------|
| **Technical Judgment** | Sees architectural implications before they become problems. Knows when AI-generated code is fragile, inefficient, or wrong. |
| **Product Judgment** | Can prioritize ruthlessly. Knows what to cut. Understands user problems deeply enough to evaluate solutions. |
| **Design Judgment** | Recognizes good UX intuitively. Can critique and improve interfaces. Understands design as function, not decoration. |
| **AI Orchestration** | Knows how to decompose problems for AI. Writes effective prompts. Manages multi-agent workflows. |
| **System Thinking** | Sees connections and dependencies. Anticipates second-order effects across technical and business domains. |
| **Commercial Awareness** | Connects work to business value. Makes trade-offs with cost/value in mind. Doesn't gold-plate. |

### Key Insight

**The constraint is Master Builders, not headcount.** You can only scale as fast as you can develop or acquire people with judgment across all six capabilities.
